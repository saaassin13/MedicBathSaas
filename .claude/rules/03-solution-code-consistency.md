# 方案与代码一致性规范

## 核心原则
任何代码变更必须先更新对应方案文档，除非变更仅是纯内部重构（不影响对外行为）。

## 强制执行步骤

### 1. 开发前
- 定位本次任务对应的方案文档（见 `docs/.project-conventions.md` → 架构设计路径）
- 在方案文档中明确标记将要修改的章节，并用 [待实现] 标注

### 2. 开发中
若发现方案不合理需要偏离，禁止直接改代码：
- 先在方案文档中更新描述（删除 [待实现] 改为 [已变更]）
- 再修改代码
- 每次偏离必须记录偏离原因到技术决策路径（见 `docs/.project-conventions.md`）

### 3. 开发完成后
- 移除所有 [待实现] / [已变更] 标记
- 在方案文档末尾添加 ## 实现检查 小节，列出：
  - 实现了哪些设计点
  - 哪些设计点因故未实现（并说明理由）
  - 新增了哪些未在设计中的功能（并说明理由）

## AI 行为检查点
在生成代码前内心确认：
1. 相关方案文档是否已阅读？
2. 本次修改是否在方案文档中有对应章节？
3. 若无对应，是否已先更新方案文档？

未通过时禁止直接输出代码。
