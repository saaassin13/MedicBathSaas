# 需求与方案对齐规范

## 目的

确保 specs（需求规格）与 plans（实施计划）完全对应，无遗漏、无歧义、无不一致。

---

## 1. 对齐分析流程

### 1.1 双向对比分析

**第一步：建立清单**
- 列出所有 spec 文档，建立清单
- 列出所有 plan 阶段，建立清单
- 制作一一对应表格

**第二步：逐项核验**

| 核验维度 | 具体检查点 |
|---------|-----------|
| 功能覆盖 | 表格列数、组件数量、筛选器数量、按钮数量 |
| 数值一致 | 颜色值（品牌色）、字号、行高 |
| 格式一致 | 时间戳格式（YYYY/MM/DD HH:mm:ss.SSS） |
| 交互一致 | 防抖时间（500ms）、筛选联动逻辑 |
| 结构对应 | spec 章节 vs plan 任务是否一一对应 |

**第三步：输出问题清单**
- 对不上的地方明确标记
- 区分优先级（高/中/低）
- 说明影响范围

---

### 2. 开发角度可行性分析

从实现视角审视计划的可行性：

| 维度 | 检查点 |
|------|--------|
| 数据流 | 筛选器是否连接到过滤逻辑？防抖是否实现？ |
| 组件复用 | 复用路径是否合理？是否会产生循环依赖？ |
| 依赖完整 | 导出功能用了 xlsx 库但 package.json 是否声明 |
| 路由结构 | 父子路由嵌套是否正确？占位符是否会覆盖子路由？ |
| 状态处理 | useState/useEffect 是否正确管理副作用 |
| 类型定义 | types/index.ts 是否包含所有使用的类型 |

---

### 3. 修正流程

发现问题后按类型执行对应流程：

| 类型 | 处理流程 |
|------|---------|
| 技术决策 | 创建 ADR 到 `docs/decisions/`，修正 plan 文件 |
| Bug 修复 | 创建 bugfixes 记录，修正 plan 文件 |
| 陷阱记录 | 创建 lessons-learned 记录，修正 plan 文件 |

**禁止行为**：
- 跳过记录流程直接修正
- 修正后不更新文档
- 只改代码不改 plan

---

### 4. 任务化管理

将发现的问题创建为任务清单，逐个完成：

```
1. 创建任务（TaskCreate）
2. 标记 in_progress
3. 执行修正
4. 标记 completed
5. 输出修改汇总
```

---

## 5. 核心原则

1. **先分析再动手** — 不急于修正，先完整理解对不上的地方
2. **分类处理** — 技术决策、bug、陷阱走不同的记录流程
3. **开发视角审视** — 对着代码思考「这里能跑起来吗」「有没有漏掉什么」
4. **逐个击破** — 问题多时用任务清单管理，完成一个是一个
5. **保持同步** — 修正计划文件后同步更新相关依赖（ADR、任务状态等）

---

## 6. AI 自检清单

完成对齐分析后，内心确认：

- [ ] 所有 spec 都有对应的 plan 阶段
- [ ] 所有 plan 阶段都有对应的 spec 依据
- [ ] 功能细节（列数、颜色、格式）已逐一核验
- [ ] 发现的问题已记录到对应文档（ADR/bugfixes/lessons-learned）
- [ ] 修正项已通过任务清单管理
- [ ] 修正完成的任务已标记 completed
